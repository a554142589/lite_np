# Benchmark Methodology

`litenp` benchmarks are meant to show two different performance stories:

1. Structure-aware lazy semantics can avoid work for arrays with known structure.
2. Materialized dense kernels can be competitive for common CPU array operations.

Those are not the same claim. The report keeps both because both matter, but row
names and documentation must make the distinction explicit.

## Categories

### Structure-Aware Fast Paths

Rows with names such as `ones`, `full`, `eye`, `transpose uniform metadata`,
`sum uniform ...`, and `matmul uniform ...` use semantic metadata. For example,
`ones<float>({n})` records shape plus the value `1`; it does not necessarily
write `n` floats until a materializing API requires storage. Likewise, reductions
or matrix products over uniform inputs may use algebraic shortcuts.

These rows measure the benefit of preserving structure. They should not be read
as generic dense memory bandwidth or generic GEMM throughput.

### Materialized Dense Kernels

Rows such as `add`, `broadcast`, `relu_into`, `sqrt_into`, `where`, `clip_into`,
and non-uniform `as_contiguous` paths operate over materialized buffers. These
are the rows closest to ordinary dense array workloads.

Allocation-free `*_into` rows time preallocated output paths. Allocating API rows
include the cost of creating the result.

## Baseline Comparisons

The benchmark manifest records which baselines are applicable for each row:

- NumPy baselines are generated by `benchmarks/bench_numpy.py`.
- Eigen and libtorch baselines are emitted by `benchmarks/bench_litenp.cpp` when
  those optional dependencies are available.
- A row with no meaningful equivalent baseline marks that baseline as disabled.

Thread-sensitive benchmark runs should pin library thread counts:

```bash
OMP_NUM_THREADS=1 OPENBLAS_NUM_THREADS=1 MKL_NUM_THREADS=1
```

The benchmark does not pin CPU frequency, isolate cores, disable turbo, or claim
cross-machine reproducibility. Treat the numbers as single-machine evidence, not
as a universal ranking.

## Reading Speedups

Large speedups in structure-aware rows usually mean `litenp` avoided
materialization or dense compute by keeping semantic metadata. That is a real
feature, but it is a different feature from a faster dense kernel.

Dense kernel rows are better evidence for common materialized workloads. They
still depend on compiler flags, vector ISA, memory bandwidth, BLAS backend,
thread settings, and cache state.

## Naming Rules

Benchmark row names should identify special structure when it drives the result:

- Use `uniform` when inputs are all the same value.
- Use `metadata` when the output can remain virtual or structure-only.
- Use `*_into` for preallocated output kernels.
- Avoid names like `materialize` unless the row forces a real dense output.

## Current Gaps

The benchmark should grow more non-uniform rows before any broad performance
claim is made. Useful additions include:

- random or patterned `sum(axis)` inputs;
- random or patterned `matmul` inputs;
- dense transpose materialization on non-uniform data;
- strided slice materialization;
- randomized broadcasting with non-uniform rows and columns.
